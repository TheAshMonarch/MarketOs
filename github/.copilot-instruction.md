# MarketOS Frontend — Project Rules

## Stack (do not substitute or add alternatives)
Next.js 16 App Router, React 19, TypeScript, Tailwind CSS, shadcn/ui, React Hook Form, Zod, Recharts.
Backend is Next.js API routes — frontend only `fetch`es `/api/*` (via `lib/api.ts`). Do not invent endpoints; do not put DB/business logic in the UI.

Fonts in use: Playfair Display (`--font-heading`) + Noto Sans (`--font-sans`). Accent token: `--brand-accent` (deep ochre). UI kit: shadcn base-sera (underline inputs, square uppercase buttons).

## Design rules — strict
- Do NOT use: cream background + terracotta/clay accent (#D97757-ish), near-black + acid-green/vermillion, or broadsheet-style hairline rules with zero border-radius. These are banned defaults.
- Palette: warm neutral/stone base (not stark white, not cream-default) + one confident accent (deep ochre already wired as `--brand-accent`).
- Typography: display face for headers, clean body face for numbers/data. No generic Inter-for-everything.
- No numbered markers (01/02/03) unless content is a genuine sequence.
- Minimal animation. Generous whitespace. Mobile-first (bottom tab nav on small screens; top nav from `md:`). Visible focus states.

## Scope — pages (implemented)
1. **Register / Login** — `/register`, `/login`. No OTP (endpoint does not exist). After login → `/dashboard`.
2. **Passport (owner)** — `/passport`. Prefill from `GET /api/passport`. Form fields: business name, owner name, market, shop number, category, photo (local preview only), description + own stats. Save is blocked until a write API exists — do not invent one in the UI.
3. **Passport (public)** — `/passport/[businessId]`, read-only. No public-by-id API yet; only works when the viewer is the logged-in owner (same `GET /api/passport` + id match). Keep public UX; do not invent `/api/business/:id` on the frontend.
4. **Smart Sale Entry** — `/sales/new`. Free-text → `POST /api/ai/parse` → editable confirmation list (name, quantity, unit; edit/remove + add-manually) → Confirm & Save loops `POST /api/sales` per line (`{ productId, quantity, paymentMethod }`). On parse failure/timeout → empty manual row (no dead-end).
5. **Dashboard** — `/dashboard`. Compose `GET /api/ai` + `GET /api/products` (there is no `/api/dashboard`). 4 stats (Revenue, Sales Today, Transactions/day-count, Products in Stock), Recharts “Weekly Sales” from `analytics.chartData`, Recent Activity list.
6. **Receipt / QR** — After sale, show receipt + client-built Google Charts QR. Public `/receipt/[receiptId]` reads query params / sessionStorage (verified, amount, seller, timestamp). No `GET /api/receipt/:id` yet.

Shared: bottom tabs / top nav for Dashboard, Passport, Sale, Logout (`proxy.ts` protects app routes). Always handle loading + error states.

## Explicitly out of scope — never generate these
Forgot-password, product edit/delete/search UI, low-stock alerts, settings, OTP/verify-otp UI, inventing `/api/business`, `/api/dashboard`, or `/api/receipt` from the client, extra AI chart pages beyond plain numbers from existing analytics.

## API contract (implemented — use only these)

```
POST /api/auth/register     body: { name, email, password, phone? }
POST /api/auth/login        body: { email, password } → httpOnly cookie `token`

GET  /api/passport          auth required → { passport: { id, businessName, market, shopNumber, category, description, reputation, ownerName, phone, qrCodeUrl } }

GET  /api/products          auth → { products: Product[] }   Product includes stock, name, price, id, …
POST /api/products          auth → { name, price, stock, category? }  (no product-management UI in scope)

POST /api/ai/parse          auth → body { text } → { items: [{ productId, matchedName, quantity }] }
POST /api/sales             auth → body { productId, quantity, paymentMethod? } → { message, sale }

GET  /api/ai                auth → { analytics: { chartData: [{ date, revenue }], totalRevenue, topProduct, advisorTip } }
```

Auth: always `credentials: "include"`. Errors return `{ error: string }`.
